# 一：背景/已有共识

长期智能体不是单轮问答系统，它的未来决策必然依赖过去交互。

过去交互包括用户偏好、历史任务、环境反馈、工具结果、失败尝试、成功轨迹、代码状态、项目约束、会话承诺等等。

完整过去不可能每次全部放进上下文，所以系统必须把过去外部化、压缩、维护、检索、调用或注入。
	这里已经有大量已有工作：
	memory 系统把过去压成 episodic records / summaries / states；
	skill 系统把成功轨迹压成 reusable procedures；
	rule / guardrail 把经验压成抽象约束；
	code agent 把项目文件、执行日志、测试结果、规范文件当作 workspace state；
	failure tracing 工作把 memory pipeline 里的变量和操作做成执行图。

>Past interaction must be externalized into reusable artifacts for long-horizon agents.

# 二：定位/问题

已有工作主要回答“过去交互如何被沉淀”。interaction traces 被压缩成什么artifact，压缩比例如何，泛化性和具体性如何权衡，生命周期如何维护。
	[Experience Compression Spectrum](https://arxiv.org/abs/2604.15877v1)借一下， _Experience Compression Spectrum_ 的主轴就是 compression axis：从 raw trace 到 episodic memory、procedural skill、declarative rule。

主要集中在回答——过去交互变成什么。

但压缩之后，未来决策仍然面临一个运行时问题：当前自然语言意图必须选择、访问、组织这些外部知识制品；如果访问接口或访问编译错误，系统即使拥有正确经验，也可能无法在当前决策中使用它。

本文定位在，处理过去交互信息，如何影响未来决策——当前意图如何访问并暴露这些信息来支持决策
	第一，memory 不是唯一对象，memory、skill、rule、workspace state 都是过去交互影响未来决策的外部化形式。
	第二，Experience Compression Spectrum 已经把这些形式统一到压缩轴上，所以本文不需要重新发明“经验”这个概念。
	第三，压缩轴之外还存在访问轴，一个artifact只有在运行时被正确访问、暴露或执行，才会正向影响当前决策。

>Given reusable artifacts compressed from past interaction, how should a current intent access them so that they can influence future agent decisions?

# 三：问题分析/数学建模

压缩轴回答：interaction trace 被压成什么 level 的 artifact。  
访问轴应该回答：当前任务通过什么 interface 使用这些 artifact。

访问接口可以只是一次性选择相关文本，也可以读取状态，也可以追踪来源，也可以遍历轨迹，也可以执行技能，也可以调用工具验证，也可以强制注入规则。

过去交互已被压缩成 reusable artifacts，不同 artifact 使用语义也不同。
	raw trace 的使用语义是回放、搜索、定位、来源追踪。  
	episodic memory 的使用语义是检索、读取、按时间或实体过滤。  
	state summary 的使用语义是查当前承诺、查当前约束、查当前项目状态。  
	procedural skill 的使用语义是调用、实例化、执行、修正。  
	declarative rule 的使用语义是注入、约束、检查、拒绝违规动作。  
	workspace artifact 的使用语义是定位、读取、编辑、测试、验证。  
	execution graph 的使用语义是追踪、归因、最小修复。

如果统一用 top-k text retrieval 访问，就会把 read、lookup、walk、execute、inject、verify 等不同语义压扁，因此压缩轴之后必须有访问轴。  
访问轴的核心是当前意图选择何种 usage interface，使过去artifact以合适方式进入当前决策。

压缩轴有 generalizability-specificity trade-off。  
访问轴也应该有自己的 trade-off性质。

可能是：
	低交互访问成本低，但只能暴露局部相关片段。  
	高交互访问成本高，但能执行、验证、追踪来源、处理冲突。  
	单次访问延迟低，但容易形成不完整视图。  
	多步访问延迟高，但能逐步补足缺失角色。  
	文本化访问通用，但会抹掉 skill/rule/tool 的使用语义。  
	原生接口访问保留使用语义，但需要更复杂的路由和控制。

## 6.9 晚上更新:

给定一组 compressed artifacts，不同访问接口暴露了不同的查询/操作语言。当前任务对应某类目标查询或目标效果。一个接口是否足够，不取决于它是否“相关”，而取决于它的访问语言能否在预算内表达并实现当前任务所需的查询或效果。

ECS 给的对象是 artifact：interaction trace 被压缩成 raw trace、episodic memory、procedural skill、declarative rule 等 reusable knowledge artifacts。它的数学是定义压缩轴 ($C_L:T\to K_L$)，再用这个轴解释系统分布和 missing diagonal。 接下去不只说 artifact 有 usage semantics，再说：**每类 artifact 诱导或要求某种 access language。**

Mnemis 就是例子。它没有空定义两个算子。它有 base graph 和 hierarchical graph 两个 artifact substrate。base graph 支持 embedding/BM25/RRF 这类 top-k similarity language；hierarchical graph 支持 top-down browsing language。它的 System-2 增益来自 hierarchical graph 允许“从类别到实体再到相关 episode/edge”的全局选择，而不是来自一个抽象“System-2 算子”。 GrepSeek 也是例子：raw corpus 如果只暴露 top-k retrieval，就是一种访问语言；如果暴露 shell pipeline，就获得 grep/filter/count/head/sort 等组合操作。Code agent 也是例子：workspace artifact 通过 read/grep/edit/test/bash 暴露，而不是普通文本块。

所以第三层:

压缩后的外部状态是一个 typed artifact store，记为 $D$。
里面有不同类型的制品：memory records、state records、traces、skills、rules、workspace files、execution logs。
一个访问接口是一种访问语言 $\mathcal L$。
例如 top-k retrieval language 只能根据 query 和 item score 返回有限个 items；
hierarchical browsing language 可以沿 category edges 选择子树；
trace language 可以按时间或来源链遍历；
skill language 可以实例化和执行 procedure；
rule language 可以注入或检查约束；
workspace language 可以定位、读取、编辑、测试和验证文件。

当前意图 $u$ 对应一个目标任务 $\phi_u(D)$。这个 $\phi_u$ 可以是事实读取、集合枚举、时间更新、来源追踪、过程执行、规则约束、验证测试。
访问语言 $\mathcal L$ 在预算 $B$ 下是否足够，可以定义为：是否存在一个访问程序 $p\in\mathcal L$，成本不超过 $B$，使得下游解码器能从 $p(D)$ 得到 $\phi_u(D)$。

>$(\mathcal L,B)$-realizability：任务是否能被某种访问语言在预算内实现。

## 6.10 早上

不能一上来就讲访问语言 $L$，那太抽象了。得先讲原语。
等 primitive trace 定义完，再说一组 primitives 和它们的组合规则可以诱导出一个 access language。这样才具备可计算/可实现性。

压缩后的外部状态可以表示为一个 typed artifact store，记为 $D$。
	其中包含不同类型的可复用制品，例如 memory records、state records、execution traces、skills、rules、workspace files 和 execution logs。每类制品不只是保存内容，还暗含一种运行时使用方式：memory record 可以被读取或检索，state record 可以被查询，trace 可以被按时间或来源链追踪，skill 可以被实例化和执行，rule 可以被注入或检查，workspace file 可以被定位、读取、编辑和测试。

我们将这些运行时使用方式形式化为 typed access primitives。设系统暴露的原语集合为 $P$。一个原语 $\rho\in P$ 可以写成  
$$
\rho=(\mathrm{dom}, \mathrm{in}, \mathrm{out}),
$$
其中 $\mathrm{dom}$ 表示该原语适用的 artifact 类型，$\mathrm{in}$ 表示输入参数类型，$\mathrm{out}$ 表示返回结果或运行时效果类型。比如，top-$k$ retrieval 作用于文本化 memory records，输入 query 和 $k$，返回 ranked items；hierarchical browse 作用于 category graph，输入 query 和当前节点，返回被选择的子节点或子树；trace primitive 作用于 ordered traces，返回时间链或来源链；skill primitive 作用于 procedural artifacts，返回执行结果；rule primitive 作用于 declarative artifacts，返回约束注入结果或合规判定；workspace primitive 作用于 files、logs 和 tests，返回读取、修改或测试结果。

一次运行时访问是一条 primitive trace：  
$$
p=((\rho_1,\theta_1,r_1),\ldots,(\rho_T,\theta_T,r_T)).
$$  
其中第 $i$ 步选择原语 $\rho_i$，给定参数 $\theta_i$，得到返回结果 $r_i$。**后续步骤的参数可以依赖之前的返回结果**。因此，primitive trace 可以表示多种真实系统中的访问过程：相似度检索、层级浏览、shell-command 搜索、文件读写测试、状态查询、trace 追踪等。

给定一组 primitives $P$，所有可由这些原语及其组合规则形成的 primitive traces 构成一个诱导的访问程序空间，记为 $\mathcal L(P)$。这里 $\mathcal L(P)$ 不需要被显式枚举；实验中实际观测和记录的是由某个访问策略产生的一条具体 trace。
> [!warning]
> 因为“组合访问程序空间”太理论化，我们无法证明整个空间的性质。真正能实验记录的是 trace，不是空间。空间 $\mathcal L(P)$ 只在定义里出现：
> $\mathcal L(P): \text{all primitive traces induced by } P$
> 然后回到实际：
> $\pi(u,h_t)\rightarrow(\rho_t,\theta_t),\quad \tau_\pi\in\mathcal L(P)$
> 

给定当前意图 $u$，它对应 artifact store 上的一个目标任务，记为 $\phi_u(D)$。这个目标可以是事实读取、集合枚举、时间更新判断、来源追踪、过程执行、规则约束或验证测试。访问 trace $p$ 在 $D$ 上执行后产生 $p(D)$，下游解码器 $g$ 基于 $p(D)$ 输出答案或动作。

为了避免把不同资源强行标量化，我们不把预算定义成单一成本函数，而是定义为一个 budget protocol，记为 $\beta$。它可以约束 primitive 调用次数、每次返回项数、最大遍历深度、最终上下文长度、工具调用轮数或延迟上限。记 $p\preceq\beta$ 表示 trace $p$ 满足该预算协议。

于是，任务 $u$ 在 $(P,\beta)$ 下可实现，当且仅当存在一条由 $P$ 中原语组成的 primitive trace $p\in\mathcal L(P)$，满足 $p\preceq\beta$，并且存在下游解码器 $g$，使得 $g(p(D))=\phi_u(D)$。

这个定义将长期记忆系统中的两个问题分开：压缩阶段决定哪些 artifacts 存在于 $D$ 中；访问阶段决定在给定原语集合 $P$ 和预算协议 $\beta$ 下，当前意图是否能够通过一条可执行 primitive trace 触达并使用这些 artifacts。换言之，命中相关 artifact 并不等价于任务可实现；任务是否可实现还取决于系统是否暴露了合适的访问原语，以及访问策略是否在预算内产生了合适的原语轨迹。

>逻辑线：usage semantics → access interface → future decision
>展开成为，artifact usage semantics → typed access primitives → primitive trace under budget → task realization $\phi_u(D)$→ decision outcome

> 压缩轴给出 artifact：过去交互被压缩成 memory、state、trace、skill、rule、workspace file 等artifact。

> 
> 决策结果回答：最后答案、动作、约束遵守、测试结果、状态更新是否正确。

#####  3：要测的部件

MMRAG SGM是从artifact store检索再回答，对应access、selection、view和decoder的失败；

NIAH SGM是gold evidence已经在候选池里但有distractors，所以测selection、view和decoder

Oracle SGM直接把gold evidence给出来，只看view和decoder；Oracle Raw是raw multimodal的上界对照，用来判断SGM是不是因为文本化丢了视觉细节。

所以完整的访问链是：$D \xrightarrow{\text{MMRAG access}} R_k \xrightarrow{\text{selection/view}} v \xrightarrow{g} y$。、

然后用NIAH和Oracle钳制这条链：
	MMRAG是$D \rightarrow R_k \rightarrow v \rightarrow y$，
	NIAH是$G_u \cup H_k \rightarrow v \rightarrow y$，
	Oracle是$G_u \rightarrow v \rightarrow y$，其中$G_u$是gold evidence，$H_k$是distractors。

实验解释就干净了：
$F_{\mathrm{MMRAG}} = F_{\mathrm{access}} + F_{\mathrm{selection}} + F_{\mathrm{view}} + F_{\mathrm{decoder}}$，
$F_{\mathrm{NIAH}} = F_{\mathrm{selection}} + F_{\mathrm{view}} + F_{\mathrm{decoder}}$，
$F_{\mathrm{Oracle}} = F_{\mathrm{view}} + F_{\mathrm{decoder}}$。

但实验还不能完全分开view和decoder。Oracle failure只能说是$F_{\mathrm{Oracle}}=F_{\mathrm{view}}+F_{\mathrm{decoder}}$。如果想进一步拆开view和decoder，就需要额外加一个轻量的view constructor，比如source-grouped view、timeline view、role-labeled view。看看$S(\text{RoleView}) - S(\text{Oracle SGM})$是否大于零。如果大于零，说明Oracle SGM的evidence已经对了，但view还不够decision-ready；如果RoleView也不提升，失败更可能是decoder、reasoning或answer ambiguity。（**这里，原论文所说的加强模型回答，是不是就是诊断view和decoder的分离？**）

**原论文通过更强回答模型测试 Oracle 条件下的回答瓶颈；我们可以进一步固定回答模型、改变 evidence view，测试 gold evidence 是已经“可决策”，还是只是“可见”。**
	原论文，固定 Oracle evidence view，换 decoder。ATM-Bench Table 4 ：在 Oracle retriever 下，Qwen3-VL-8B Hard QS 是 47.3，GPT-5 提到 74.7；论文把这一节直接叫 “Identifying Bottlenecks in Answer Generation”。论文自己也说，即使给了 ground-truth memory items，Hard 仍然很难，可能原因是需要聚合和 reconcile 更多 memory items。
		 我们可以进一步研究 view/decoder 混合区：aggregation / reconciliation 可以是 decoder 能力不足，也可以是 view 没把 source、time、conflict、aggregation role 暴露清楚。**同一个模型，如果换 evidence view，能不能更好？**

### 先收束一下故事线：Evidence View Realization

一个长时程智能体将过去的交互存储为一个带类型的artifact存储 $D$。

该存储可能包含记忆记录、状态记录、轨迹、规则、技能、工作空间文件、日志或其他可复用的制品。

它们必须通过系统暴露的运行时接口访问，并序列化为下游模型实际可用的视图。

设 $P$ 为记忆系统暴露的一组带类型的访问原语。一个原语 $\rho \in P$ 指定了其可操作的制品类型、输入参数以及返回的观察结果或运行时效果。示例包括对记忆卡的检索、结构化状态查找、图浏览、源代码追踪、文件读取、规则注入和测试执行。

给定一个意图 $u$，一个访问策略 $\pi$ 生成一个原语轨迹
$$
p_\pi(u,D)=((\rho_1,\theta_1,r_1),\ldots,(\rho_T,\theta_T,r_T)),
$$
其中每一步选择一个原语 $\rho_t$，提供参数 $\theta_t$，并获得结果 $r_t$。该轨迹必须满足一个预算协议 $\beta$，该协议可能限制调用次数、返回项数、遍历深度、上下文长度、延迟或工具使用轮次。

轨迹返回的信息随后被序列化为一个证据视图：
$$
v = V(u,p_\pi(D)).
$$

下游模型 $g$ 接收当前意图、可选上下文 $c$ 以及证据视图：
$$
\hat y = g(u,c,v).
$$

分析的核心对象并非相关制品是否存在于 $D$ 中，也非检索到的项与查询的相似度高低。核心对象是：原语轨迹是否暴露了一个对决策充分的证据视图。

对于每个任务 $u$，定义一组所需的视图角色：
$$
R_u=\{r_1,\ldots,r_k\}.
$$
这些角色描述了在当前决策得到支持时，证据视图中必须可观察到的内容。在个人记忆问答中，典型角色包括证据可见性、源身份、时间定位、空间定位、指代绑定、多证据聚合、冲突权威性、时效性和弃权支持。一个视图 $v$ 暴露了角色 $r$，如果该角色出现在模型可见的内容中，而不仅仅是内部存储。令$\Gamma(v)$表示 $v$ 暴露的角色集合。当$R_u \subseteq \Gamma(v)$时，视图对于任务 $u$ 是充分的。

这给出了一个视图级别的失败分解。如果所需artifact在 $D$ 中缺失，则为基底失败。如果artifact存在但没有任何预算可行的原语轨迹能暴露它，则为访问失败。如果轨迹暴露了相关证据，但序列化视图缺少必要的绑定、权威关系、聚合结构或定位线索，则为视图构建失败。如果视图暴露了所有所需角色而模型仍然回答错误，则将失败归因于下游解码或推理。

这一区分将存储、访问、视图构建和解码分开：
$$
D \xrightarrow{p_\pi} p_\pi(D) \xrightarrow{V} v \xrightarrow{g} \hat y.
$$
因此，同一artifact存储可以在不同的原语集、预算、轨迹或视图构造器下支持不同的结果。相反，两个系统可能检索到相同的证据项，但最终视图是否暴露了任务所需的角色可能不同。

在 ATM-Bench 中，这一分解可以通过三种官方证据设置来操作化。
	Oracle 视图使用注释的证据 ID，在完美证据可见性下隔离视图构建和解码。NIAH 视图保证oracle证据存在于干扰项中，测试模型是否能在上下文噪声中选出并组合证据。基于检索的视图使用系统实际的访问方法，测试原语轨迹是否在现实预算下暴露了所需证据。

设 $S_\theta(V)$ 为模型 $g_\theta$ 在视图构造器 $V$ 下的平均任务得分。我们可以估计三个差距：
$$
G_{\mathrm{access}} = S_\theta(V_{\mathrm{NIAH}}) - S_\theta(V_{\mathrm{retrieval}}),
$$
$$
G_{\mathrm{selection}} = S_\theta(V_{\mathrm{oracle}}) - S_\theta(V_{\mathrm{NIAH}}),
$$
$$
G_{\mathrm{view}} = S_\theta(V_{\mathrm{role}}) - S_\theta(V_{\mathrm{oracle}}).
$$
第一个差距衡量访问方法是否暴露了所需证据。第二个衡量模型是否能在干扰项中识别并组合黄金证据。第三个衡量官方的黄金证据视图本身是否已经对决策充分，或者额外的角色暴露是否能改进决策。

>这一框架避免了将记忆访问视为简单的 top-$k$ 检索。检索到的项仅仅是访问轨迹可能的返回值之一。对最终决策重要的是该轨迹诱导出的证据视图，以及该视图是否暴露了当前意图所需的所有角色。

>集合语言比加法表示更加严谨完备,后期可用再修一下。不过写不写还是得看实验

### STALE

STALE把长期记忆问题收窄到一个很具体的能力：**新观察隐式使旧记忆失效时，模型能不能知道旧记忆已经不该继续支配行为**。论文称之为 **Implicit Conflict**：later observation invalidates an earlier memory without explicit negation。

它构造了 400 个 expert-validated conflict scenarios，每个 scenario 有 3 个 probing queries，总共 1,200 个查询；三类 probe 是 State Resolution、Premise Resistance、Implicit Policy Adaptation。 arXiv 摘要也确认它的目标是测试 belief revision，而不是静态 fact retrieval。
https://github.com/icedreamc/STALE

###### 论文思路：
STALE 把"前提污染"（PR 维度）单独拎出来，这在 memory QA 里是常被忽略的维度
	模型能拿到 updated evidence 不等于它会在用户用过期前提提问时拒绝回答。
	LightMem 和 mem-0 的 SR/PR 差距说明 retrieval is not enough——你 retrieve 到了对的，但模型还是顺着用户的前提走了。

这已经把 retrieval is not enough 说死了：新证据被存了、也被取到了，但没有变成回答的支配依据。 这点和我们现在的主线高度相关，**它也观察到了retrieval visibility 与 behavioral use 之间有断层。**

CUPMem 是有效但不够通用的解法。它的三件事是 write-side belief updating、topology-triggered belief propagation、authorized readout；

 `cup_mem/` 包含 structured memory write/invalidation、retrieval and premise verification、conflict-aware readout。

这套机制很有针对性，所以分数从 GPT-4o-mini 的 8.7% 提到 68.0%，PR 提升尤其大。但它的代价是 schema dependence：论文 limitation 明说 CUPMem 是 targeted prototype，依赖 predefined state schema，不是 general-purpose memory architecture；schema-free evolving user state 仍没解决。
	**不过也正是我们想说的，一个子任务真正被系统化解决，往往就是先抽出合适的 artifact substrate，再定义一组能操作这个 substrate 的 primitives。STALE/CUPMem 正好是这个模式**

> 补充：
> 之前这样的叙事会导致我们失去artifact->primitive的解释权
> *General access layer:*
> $\text{compression} \rightarrow \text{artifact} \rightarrow \text{usage semantics} \rightarrow \text{typed primitives} \rightarrow \text{primitive trace}$
> *This paper’s operational layer:*
> $\text{primitive trace} \rightarrow \text{evidence view} \rightarrow \text{decision support}$
> 
> $\text{primitive trace over artifacts}→\text{decision-facing artifact/view state}$会比较好收纳STALE


然后 STALE 可以被自然解释成一个子任务实例：
$\text{Subtask} = \text{stale-state validity}$   $\text{STALE——STATE}$
$\text{Artifact substrate} = D^{\text{state}}$  
$P_{\text{STALE}} = \{ \rho_{\text{extract}}, \rho_{\text{propagate}},\rho_{\text{adjudicate}}, \rho_{\text{authorize}}, \rho_{\text{readout}} \}$
它做的是：$D_t \rightarrow D_t^{\text{auth}} \rightarrow v_u^{\text{state}} \rightarrow y$
**证明在 state-validity 子任务上，正确做法是先设计 state-temporal artifact，再配套一组能维护和读取 current validity 的 primitives。**

用得还不够，不能只是解释，我们的贡献点要有。比如把它的下游做出来？这样就收敛到我们的position了。

可以把它当成研究 primitive-role alignment 的受控底座。CUPMem 证明了 state-validity 失败可以通过特化 primitive stack 缓解；我们进一步拆解这个 stack 与 SR/PR/IPA 三类下游 probe 的关系，分析哪些原语分别支持状态识别、前提抵抗和行为适应。
>这样感觉可能有一些外扩。

下游的话，也可以回到ATM-BENCH类的view-decision分析，validate对上一个benchmark给出的解法，这样可能整体更加集中，更加solid，把一个局部做稳定比全做“各吃一块”要好一些吧。
当然理想的话，就是这个做state artifact→primitive stack→SR/PR/IPA，且做retrieval visibility 与 behavioral use 之间断层的下游view_decision解释。

> [!question]
> 长期记忆失败不能只归因于检索失败，通过MMRAG、NIAH、Oracle的受控分层，可以把access failure、selection failure、view和decoder failure分离出来，这个是ATM-BENCH实验负责的。
> 
> STALE 负责证明一种具体 memory-use subtask 需要特化 artifact 和特化 primitives，同时也看看能不能试图验证全链条优化成立吧，如果可以，就更加有说服力，不然好像我们只能做分离分析归因，给不了综合解法似的。

整篇的逻辑肯定还是分析层 + 解法层嘛：

**ATM-Bench 负责分析层。**  
它有 MMRAG / NIAH / Oracle 这种天然分层，适合证明长期记忆失败不只是 retrieval miss。
所以它负责回答：$\text{failure comes from where?}$ 
也就是 access、selection、view/decoder 的条件分离。

**STALE 负责解法层。**  
STALE 更窄，专门测 implicit conflict / stale state：400 个场景、1,200 个 query、SR/PR/IPA 三维 probe，并指出系统常出现“retrieved updated evidence but failed to act on it”的断层；CUPMem 则通过 structured state consolidation、propagation-aware search、write-time revision 做出明显提升。

If we know the failure is not merely retrieval, what kind of artifact + primitive stack can fix a concrete subtask?


### Mnemis

虽然它几乎刷爆了旧benchmark，但是还是可以分析没满分的子任务的?

>它只留了gold evidence,不好分析。
#### 优势：

| 维度         | 分数    | 说明                     |
| ---------- | ----- | ---------------------- |
| SSP (偏好提取) | 100.0 | 单 session 偏好，几乎饱和      |
| SSA (助手信息) | 100.0 | 同上                     |
| SSU (用户信息) | 98.6  | 接近饱和                   |
| Single-Hop | 96.2  | 单条证据直查                 |
| KU (知识更新)  | 93.6  | 检测旧信息是否过期              |
| Multi-Hop  | 91.8  | 跨多证据聚合，Mnemis 相对基线差距最大 |

这些都是"找到证据就行"的任务。Single-Hop 和 SSU/SSA 全在 95+，说明 Mnemis 的基础检索精度已经没有问题。
#### 弱势

| 维度 | 分数 | 问题 |
| --- | --- | --- |
| Open-Domain (开放性推理) | 82.3 | 最低。这不是检索问题，是推理问题——找全证据只是前提，还需要判断/预测 |
| Temporal (时间推理) | 90.3 | 论文明确说 System-2 在时间类上弱，因为遍历路径不变，抓不到完整时序链 |
| MS (跨 session) | 86.5 | LongMemEval-S 上最低之一，跨 session 聚合仍有遗漏 |
| TR (时间推理) | 86.5 | 同上 |

关键区别：Weak = 检索到了但推理不够，还是漏了关键证据？ Open-Domain 的 82 分说明前者（检索够了但推理受限），而 Temporal 的 90 和 MS 的 86 说明后者（System-2 的静态路径没覆盖到时序线索）。

System-2 带来的提升（消融实验）
论文 Table 4 + 分析给出了确切的贡献拆解：、

| 系统配置 | 效果 |
| --- | --- |
| System-1 RAG (仅 episodes) | baseline |
| System-1 Graph (仅 entities+edges) | 略高 ← 更浓缩 |
| System-1 RAG + Graph | 更高 ← 互补 |
| System-1 Re-ranked | 持平 ← 提升来自 reranker 不是本质 |
| System-2 Only | 可用，但只有 ~90% 查询能用 |
| System-1 + System-2 | **全部类别提升** ← 真正补了 coverage |

System-2 带来的增益主要体现在：
1. Multi-Hop 和 Open-Domain — 这两个类别靠纯相似度容易漏（相似度最匹配的不等于最相关的）。System-2 的层级浏览能找到间接关联的 evidence
2. Coverage gap 填补 — 论文说有 90.06% 的查询 System-2 能贡献有效结果，说明纯 System-1 在约 10% 的查询上存在覆盖盲区
3. 具体 case — gastritis vs weight 那个例子最典型：System-1 搜到表面病因（胃炎），System-2 通过层级浏览找到根本原因（超重）

System-2 不擅长的：
- Temporal 类 — 因为浏览路径在遍历过程中不变，抓不住完整的时序链。论文原文也承认这一点
- 需要多次搜索的场景 — 当前 System-2 是一次性的自上而下遍历，不能像 multi-step reasoning 那样根据中间结果调整路径

System-2 补的是 coverage 和 recall，不是 precision。Mnemis 赢在"找得更全"，不是"找得更准"。

>层级浏览能扩大候选证据覆盖，但最终答案仍取决于哪些浏览结果被保留下来并序列化进最终视图。
