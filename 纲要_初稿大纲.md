## 0. 当前论文定位

长期智能体不是单轮问答系统，它的未来决策必然依赖过去交互。

过去交互包括用户偏好、历史任务、环境反馈、工具结果、失败尝试、成功轨迹、代码状态、项目约束、会话承诺等等。完整过去不可能每次全部放进上下文，所以系统必须把过去外部化、压缩、维护、检索、调用或注入。

已有工作主要回答“过去交互如何被沉淀”。Interaction traces 被压缩成什么 artifact，压缩比例如何，泛化性和具体性如何权衡，生命周期如何维护。Experience Compression Spectrum 可以作为背景：它的主轴是 compression axis，从 raw trace 到 episodic memory、procedural skill、declarative rule。

本文定位在压缩之后的运行时问题：

> 处理过去交互信息如何影响未来决策。当前意图如何访问并暴露这些信息来支持决策。

核心问题：

> Given reusable artifacts compressed from past interaction, how should a current intent access them so that they can influence future agent decisions?

需要强调三点：

1. Memory 不是唯一对象。Memory、skill、rule、workspace state、execution trace 都是过去交互影响未来决策的外部化形式。
2. Experience Compression Spectrum 已经把这些形式统一到压缩轴上，所以本文不重新发明“经验”概念。
3. 压缩轴之外还存在访问轴。一个 artifact 只有在运行时被正确访问、暴露或执行，才会正向影响当前决策。

## 1. 论文主张的最小版本

本文不应声称提出完整的 general-purpose long-horizon memory architecture。

更稳的主张是：

> We first define a general primitive-based access model for typed interaction artifacts. We then focus on the information-exposure slice of this model: how a current intent constructs an evidence view or state view over past interaction artifacts, and how this view supports downstream decisions.

中文版本：

> 本文先区分 artifact availability、evidence visibility 和 decision use，再用 ATM-Bench 的受控分层观察 access、selection、view / decoder failure，用 STALE 这样的 state-validity 子任务讨论什么时候需要特化 artifact substrate 和 primitive stack。最终方法贡献是否成立，取决于 view re-layout 或 authorized state readout 是否能在实验中把“可见证据”变成“支配决策的证据”。

本文当前贡献边界：

- 上位建模保持一般性。
- 实证和操作层主动收敛到信息暴露切片。
- 不研究所有 primitive 的完整 runtime effect。
- 不解决 full skill execution、rule enforcement、workspace editing。
- 重点研究从 \(D\) 到当前决策输入的 evidence view construction / state view construction。

## 2. 缺口：从压缩轴到访问轴

压缩轴回答：

> Interaction trace 被压成什么 level 的 artifact。

访问轴回答：

> 当前任务通过什么 interface 使用这些 artifact。

不同 artifact 自带不同使用语义：

- Raw trace 的使用语义是回放、搜索、定位、来源追踪。
- Episodic memory 的使用语义是检索、读取、按时间或实体过滤。
- State summary 的使用语义是查当前承诺、查当前约束、查当前项目状态。
- Procedural skill 的使用语义是调用、实例化、执行、修正。
- Declarative rule 的使用语义是注入、约束、检查、拒绝违规动作。
- Workspace artifact 的使用语义是定位、读取、编辑、测试、验证。
- Execution graph 的使用语义是追踪、归因、最小修复。

如果统一用 top-k text retrieval 访问，就会把 read、lookup、walk、execute、inject、verify 等不同语义压扁。因此，多路访问不是“多做几种 retrieval”，而是不同 artifact 不能用同一个访问接口抹平。

更准确的缺口不是“top-k retrieval 不够”，而是：

> Artifact availability、evidence visibility 和 decision use 之间没有被清楚地区分。

现有系统可以讨论 artifact 如何形成、如何召回、如何暴露，但较少分析 artifact 如何获得当前决策中的作用地位。这个作用地位在 ATM-Bench 中表现为 referential evidence 是否被选中、组合和解释；在 STALE 中表现为 updated state 是否压过 stale premise 并支配后续行动。

## 3. 理论线：artifact usage semantics 到 primitive trace

### 3.1 访问语言与可实现性

一个接口是否足够，不取决于它是否“相关”，而取决于它的访问语言能否在预算内表达并实现当前任务所需的查询或效果。

不同 artifact substrate 诱导或要求不同 access language：

- Top-k retrieval language 只能根据 query 和 item score 返回有限个 items。
- Hierarchical browsing language 可以沿 category edges 选择子树。
- Trace language 可以按时间或来源链遍历。
- Skill language 可以实例化和执行 procedure。
- Rule language 可以注入或检查约束。
- Workspace language 可以定位、读取、编辑、测试和验证文件。

当前意图 \(u\) 对应 artifact store 上的目标任务 \(\phi_u(D)\)。这个目标可以是事实读取、集合枚举、时间更新判断、来源追踪、过程执行、规则约束或验证测试。

访问语言 \(\mathcal L\) 在预算 \(B\) 下是否足够，可以定义为：是否存在一个访问程序 \(p \in \mathcal L\)，成本不超过 \(B\)，使得下游解码器能从 \(p(D)\) 得到 \(\phi_u(D)\)。

可以使用的概念：

> \((\mathcal L,B)\)-realizability：任务是否能被某种访问语言在预算内实现。

后文更适合改写为 \((P,\beta)\)-realizability，因为 primitives 比抽象 access language 更可计算、可实现。

### 3.2 Typed artifact store

压缩后的外部状态表示为 typed artifact store，记为 \(D\)。

\(D\) 包含不同类型的可复用制品，例如 memory records、state records、execution traces、skills、rules、workspace files 和 execution logs。

每类制品不只是保存内容，还暗含一种运行时使用方式。Memory record 可以被读取或检索，state record 可以被查询，trace 可以被按时间或来源链追踪，skill 可以被实例化和执行，rule 可以被注入或检查，workspace file 可以被定位、读取、编辑和测试。

### 3.3 Typed access primitives

设系统暴露的原语集合为 \(P\)。一个原语 \(\rho \in P\) 可以写成：

\[
\rho=(\mathrm{dom}, \mathrm{in}, \mathrm{out}),
\]

其中：

- \(\mathrm{dom}\) 表示该原语适用的 artifact 类型；
- \(\mathrm{in}\) 表示输入参数类型；
- \(\mathrm{out}\) 表示返回结果或运行时效果类型。

示例：

- Top-\(k\) retrieval 作用于文本化 memory records，输入 query 和 \(k\)，返回 ranked items。
- Hierarchical browse 作用于 category graph，输入 query 和当前节点，返回被选择的子节点或子树。
- Trace primitive 作用于 ordered traces，返回时间链或来源链。
- Skill primitive 作用于 procedural artifacts，返回执行结果。
- Rule primitive 作用于 declarative artifacts，返回约束注入结果或合规判定。
- Workspace primitive 作用于 files、logs 和 tests，返回读取、修改或测试结果。

### 3.4 Primitive trace

一次运行时访问是一条 primitive trace：

\[
p=((\rho_1,\theta_1,r_1),\ldots,(\rho_T,\theta_T,r_T)).
\]

其中第 \(i\) 步选择原语 \(\rho_i\)，给定参数 \(\theta_i\)，得到返回结果 \(r_i\)。后续步骤的参数可以依赖之前的返回结果。

因此，primitive trace 可以表示多种真实系统中的访问过程：相似度检索、层级浏览、shell-command 搜索、文件读写测试、状态查询、trace 追踪等。

给定一组 primitives \(P\)，所有可由这些原语及其组合规则形成的 primitive traces 构成一个诱导的访问程序空间，记为 \(\mathcal L(P)\)。

注意：\(\mathcal L(P)\) 不需要被显式枚举。实验中实际观测和记录的是由某个访问策略产生的一条具体 trace。

\[
\pi(u,h_t)\rightarrow(\rho_t,\theta_t),\quad \tau_\pi\in\mathcal L(P)
\]

### 3.5 Budget protocol

为了避免把不同资源强行标量化，不把预算定义成单一成本函数，而定义为一个 budget protocol，记为 \(\beta\)。

\(\beta\) 可以约束：

- primitive 调用次数；
- 每次返回项数；
- 最大遍历深度；
- 最终上下文长度；
- 工具调用轮数；
- 延迟上限。

记 \(p\preceq\beta\) 表示 trace \(p\) 满足该预算协议。

### 3.6 \((P,\beta)\)-realizability

给定当前意图 \(u\)，它对应 artifact store 上的目标任务 \(\phi_u(D)\)。

访问 trace \(p\) 在 \(D\) 上执行后产生 \(p(D)\)，下游解码器 \(g\) 基于 \(p(D)\) 输出答案或动作。

任务 \(u\) 在 \((P,\beta)\) 下可实现，当且仅当存在一条由 \(P\) 中原语组成的 primitive trace \(p\in\mathcal L(P)\)，满足 \(p\preceq\beta\)，并且存在下游解码器 \(g\)，使得：

\[
g(p(D))=\phi_u(D).
\]

这个定义将长期记忆系统中的两个问题分开：

- 压缩阶段决定哪些 artifacts 存在于 \(D\) 中；
- 访问阶段决定在给定原语集合 \(P\) 和预算协议 \(\beta\) 下，当前意图是否能够通过一条可执行 primitive trace 触达并使用这些 artifacts。

换言之，命中相关 artifact 并不等价于任务可实现。任务是否可实现还取决于系统是否暴露了合适的访问原语，以及访问策略是否在预算内产生了合适的原语轨迹。

### 3.7 逻辑线

完整理论线：

\[
\text{compression} \rightarrow \text{artifact} \rightarrow \text{usage semantics} \rightarrow \text{typed primitives} \rightarrow \text{primitive trace under budget} \rightarrow \text{task realization} \rightarrow \text{decision outcome}
\]

更适合写在论文中的 general access layer 是：

\[
\text{compression} \rightarrow \text{artifact} \rightarrow \text{usage semantics} \rightarrow \text{typed primitives} \rightarrow \text{primitive trace}
\]

这一层回答：过去交互如何从压缩制品，变成当前任务可以实际调用的一条访问轨迹。

但本文操作层不研究所有 primitive 的完整 runtime effect，而收敛为：

\[
\text{primitive trace over artifacts} \rightarrow \text{decision-facing artifact/view state}
\]

这句话比 “primitive trace \(\rightarrow\) evidence view” 更稳。ATM-Bench 中，decision-facing state 主要表现为 evidence view；STALE 中，它表现为 authorized state view 或 current-validity view。

进一步到下游任务时，才写成：

\[
\text{decision-facing artifact/view state} \rightarrow \text{decision support}
\]

## 4. Evidence View Realization

### 4.1 收敛范围

本文不研究所有 primitive 的完整 runtime effect，而研究其中与过去交互信息如何暴露给当前决策有关的部分。

也就是说，本文关注的是从 \(D\) 到当前决策输入的 evidence view construction，而不是 skill execution、rule enforcement、workspace editing 的完整闭环。

General access layer:

\[
\text{compression} \rightarrow \text{artifact} \rightarrow \text{usage semantics} \rightarrow \text{typed primitives} \rightarrow \text{primitive trace}
\]

This paper's operational layer:

\[
\text{primitive trace over artifacts} \rightarrow \text{decision-facing artifact/view state} \rightarrow \text{decision support}
\]

这里的 decision-facing artifact/view state 是一个统称。对 personalized memory QA，它通常是 evidence view；对 stale-state validity，它是带 current authority 的 state view；对其他 agent 场景，它也可能是 rule-constrained context、trace-backed provenance view 或 workspace verification state。本文实证只覆盖前两类。

### 4.2 Evidence view

一个长时程智能体将过去的交互存储为一个带类型的 artifact 存储 \(D\)。

这些 artifacts 必须通过系统暴露的运行时接口访问，并序列化为下游模型实际可用的视图。

给定一个意图 \(u\)，一个访问策略 \(\pi\) 生成一个原语轨迹：

\[
p_\pi(u,D)=((\rho_1,\theta_1,r_1),\ldots,(\rho_T,\theta_T,r_T)).
\]

该轨迹必须满足预算协议 \(\beta\)。

轨迹返回的信息随后被序列化为一个证据视图：

\[
v = V(u,p_\pi(D)).
\]

下游模型 \(g\) 接收当前意图、可选上下文 \(c\) 以及证据视图：

\[
\hat y = g(u,c,v).
\]

分析的核心对象并非相关制品是否存在于 \(D\) 中，也非检索到的项与查询的相似度高低。核心对象是：

> 原语轨迹是否暴露了一个对决策充分的证据视图。

为了覆盖 STALE 这类状态有效性任务，可以把上式更一般地写成：

\[
s_u = A(u,p_\pi(D)),
\]

其中 \(s_u\) 是由 primitive trace over artifacts 产生的 decision-facing artifact/view state。ATM-Bench 中 \(s_u\) 主要是证据视图 \(v\)；STALE 中 \(s_u\) 是 current-authority-aware state view，例如 \(D_t^{\text{auth}}\) 或 \(v_u^{\text{state}}\)。

### 4.3 Required view roles

对于每个任务 \(u\)，定义一组所需的视图角色：

\[
R_u=\{r_1,\ldots,r_k\}.
\]

这些角色描述了在当前决策得到支持时，证据视图中必须可观察到的内容。

在个人记忆问答中，典型角色包括：

- 证据可见性；
- 源身份；
- 时间定位；
- 空间定位；
- 指代绑定；
- 多证据聚合；
- 冲突权威性；
- 时效性；
- 弃权支持。

一个视图 \(v\) 暴露了角色 \(r\)，如果该角色出现在模型可见的内容中，而不仅仅是内部存储。

令 \(\Gamma(v)\) 表示 \(v\) 暴露的角色集合。当：

\[
R_u \subseteq \Gamma(v)
\]

时，视图对于任务 \(u\) 是充分的。

### 4.4 Failure decomposition

这给出了一个视图级别的失败分解：

- 如果所需 artifact 在 \(D\) 中缺失，则为基底失败。
- 如果 artifact 存在但没有任何预算可行的原语轨迹能暴露它，则为访问失败。
- 如果 trace 暴露了相关证据，但序列化 view 缺少必要的绑定、权威关系、聚合结构或定位线索，则为视图构建失败。
- 如果 view 暴露了所有所需角色而模型仍然回答错误，则将失败归因于下游解码或推理。

这一区分将存储、访问、视图构建和解码分开：

\[
D \xrightarrow{p_\pi} p_\pi(D) \xrightarrow{V} v \xrightarrow{g} \hat y.
\]

因此，同一 artifact 存储可以在不同的原语集、预算、轨迹或视图构造器下支持不同的结果。相反，两个系统可能检索到相同的证据项，但最终视图是否暴露了任务所需的角色可能不同。

注意：后文的 failure decomposition 不宜写成严格可加的独立失败项。更稳的表述是“实验条件下的诊断区间”或“逐层移除困难源的条件分解”。

## 5. ATM-Bench 实验线：分析层

### 5.1 为什么用 ATM-Bench

ATM-Bench 的出发点是：现有长期记忆 benchmark 多数还在 dialogue history 里做 QA，但真实个人记忆不是这样。真实记忆来自 photos、videos、emails、metadata，而且很多问题不是直接问“某条事实是什么”，而是问一个和个人经历绑定的 referential question。

例子：“the gift I bought for mum during our trip to Japan”。这类问题需要把用户表达里的 personal reference 落到过去生活轨迹里的具体 memory items。

ATM-Bench 适合负责本文的分析层。它有 MMRAG / NIAH / Oracle 这种天然分层，适合回答：

\[
\text{failure comes from where?}
\]

也就是 access、selection、view / decoder 的条件分离。

### 5.2 官方证据条件

同一组 QA 被三类实验以不同方式使用 evidence IDs。

标准端到端：

- 模型：query \(\rightarrow\) 检索 \(\rightarrow\) top-k evidence \(\rightarrow\) answer。
- 评估：拿 evidence_ids 对比检索结果，算 R@10。
- 推理时 evidence_ids 不给模型。

Oracle：

- 模型：query + evidence_ids 对应的黄金证据 \(\rightarrow\) answer。
- 评估：只算 answer QS。
- Evidence_ids 直接送进 prompt。

NIAH：

- 模型：query + niah_evidence_ids 对应的证据池（gold + distractors）\(\rightarrow\) answer。
- 评估：只算 answer QS，但受 distractor 密度影响。

从诊断层级看：

\[
\text{Oracle} \rightarrow \text{NIAH} \rightarrow \text{标准端到端}
\]

逐层加难度：

\[
\text{回答} \rightarrow \text{回答 + 干扰物} \rightarrow \text{检索 + 回答}
\]

### 5.3 数据集选择

主数据集先用 ATM-Bench-Hard。

理由：

- Hard 只有 31 题，适合人工精读、错误归因、做 view contrast。
- README 当前 Oracle / NIAH snapshot 也主要在 31-question Hard split 上报告。
- 全量 ATM-Bench 可以后面验证趋势，不用一开始做机制分析。

### 5.4 模型与评判

模型分四类角色：

Memory processor：

- 固定用官方产物，直接用 `output/image/qwen3vl2b/batch_results.json` 和 `output/video/qwen3vl2b/batch_results.json`。
- 完整下载会提供 preprocessed memory、emails、raw images/videos 和 GPS cache，并 staged 到脚本预期目录。
- SGM 的 image/video fields 包括 time、location、entities、ocr、tags；email 包括 time、summary、body。

Retriever：

- Embedding 先用 all-MiniLM-L6-v2。

主回答模型：

- 选 Qwen3-VL-8B-Instruct-FP8。
- 这是官方 memory-system baseline 里的固定 answerer，适合对齐 repo。

强回答模型：

- 再选一个 DS / GPT / Claude / GLM。
- 强模型只需要跑 Oracle SGM 和 NIAH SGM，用于判断“是不是模型太弱”。

评判模型：

- 沿用 gpt-5-mini。
- README 里 agent results 和 Oracle / NIAH snapshot 都标明 QS 使用 gpt-5-mini judge。
- 后面可加 GLM judge 做一致性检查。

### 5.5 要测的部件

MMRAG SGM：

- 从 artifact store 检索再回答。
- 对应 access、selection、view 和 decoder 的失败。
- 重点看 Recall@kGT 和 retrieval details，不只看 QS。

NIAH SGM：

- Gold evidence 已经在候选池里但有 distractors。
- 测 selection、composition、view 和 decoder。

Oracle SGM：

- 直接把 gold evidence 给出来。
- 没有检索，没有候选选择。
- 测 gold-evidence utilization，也就是 view 和 decoder。

Oracle Raw：

- Raw multimodal 的上界对照。
- 判断 SGM 是否因为文本化损失视觉细节。
- 也要注意：Raw 在 haystack 和 realistic 条件下可能因 distractors、payload 和 context limits 崩掉，SGM 更稳。

### 5.6 访问链与条件钳制

完整访问链：

\[
D \xrightarrow{\text{MMRAG access}} R_k \xrightarrow{\text{selection/view}} v \xrightarrow{g} y.
\]

用 NIAH 和 Oracle 钳制这条链：

\[
\text{MMRAG}: D \rightarrow R_k \rightarrow v \rightarrow y
\]

\[
\text{NIAH}: G_u \cup H_k \rightarrow v \rightarrow y
\]

\[
\text{Oracle}: G_u \rightarrow v \rightarrow y
\]

其中 \(G_u\) 是 gold evidence，\(H_k\) 是 distractors。

可作为诊断近似的失败区间：

\[
F_{\mathrm{MMRAG}} \approx F_{\mathrm{access}} + F_{\mathrm{selection}} + F_{\mathrm{view}} + F_{\mathrm{decoder}}
\]

\[
F_{\mathrm{NIAH}} \approx F_{\mathrm{selection}} + F_{\mathrm{view}} + F_{\mathrm{decoder}}
\]

\[
F_{\mathrm{Oracle}} \approx F_{\mathrm{view}} + F_{\mathrm{decoder}}
\]

注意：这里不要声称严格可加或独立，只作为条件分层解释。

### 5.7 Gap estimates

设 \(S_\theta(V)\) 为模型 \(g_\theta\) 在视图构造器 \(V\) 下的平均任务得分。

可以估计三个差距：

\[
G_{\mathrm{access}} = S_\theta(V_{\mathrm{NIAH}}) - S_\theta(V_{\mathrm{retrieval}})
\]

\[
G_{\mathrm{selection}} = S_\theta(V_{\mathrm{oracle}}) - S_\theta(V_{\mathrm{NIAH}})
\]

\[
G_{\mathrm{view}} = S_\theta(V_{\mathrm{role}}) - S_\theta(V_{\mathrm{oracle}})
\]

解释：

- \(G_{\mathrm{access}}\)：访问方法是否暴露了所需证据。
- \(G_{\mathrm{selection}}\)：模型是否能在干扰项中识别并组合黄金证据。
- \(G_{\mathrm{view}}\)：官方黄金证据视图本身是否已经对决策充分，或者额外的角色暴露是否能改进决策。

### 5.8 View / decoder 混合区

Oracle failure 不能完全分开 view 和 decoder。Oracle failure 只能说：

\[
F_{\mathrm{Oracle}}=F_{\mathrm{view}}+F_{\mathrm{decoder}}
\]

如果想进一步拆开 view 和 decoder，需要额外加一个轻量 view constructor，例如：

- source-grouped view；
- timeline view；
- role-labeled view。

只在 Oracle 错例上做，不当作大方法，用来分开 view failure 和 decoder failure。

判断：

- 如果 \(S(\text{RoleView}) - S(\text{Oracle SGM}) > 0\)，说明 Oracle SGM 的 evidence 已经对了，但 view 还不够 decision-ready。
- 如果 RoleView 也不提升，失败更可能是 decoder、reasoning 或 answer ambiguity。

ATM-Bench 原论文通过更强回答模型测试 Oracle 条件下的回答瓶颈。本文可以进一步固定回答模型、改变 evidence view，测试 gold evidence 是已经“可决策”，还是只是“可见”。

## 6. STALE 实验线：state-validity case study

### 6.1 为什么用 STALE

STALE 把长期记忆问题收窄到一个很具体的能力：

> 新观察隐式使旧记忆失效时，模型能不能知道旧记忆已经不该继续支配行为。

论文称之为 Implicit Conflict：later observation invalidates an earlier memory without explicit negation。

STALE 构造了 400 个 expert-validated conflict scenarios。每个 scenario 有 3 个 probing queries，总共 1,200 个查询。

三类 probe：

- State Resolution；
- Premise Resistance；
- Implicit Policy Adaptation。

它的目标是测试 belief revision，而不是静态 fact retrieval。

### 6.2 数据和任务结构

数据集：

- 400 个 expert-validated implicit conflict scenarios。
- 100+ 日常生活主题。
- 每个 scenario 包含 \(M_{\mathrm{old}}\)、\(M_{\mathrm{new}}\)、explanation、probing queries、timestamps、haystack sessions。

冲突类型：

Type I: Co-referential Implicit Conflict。

- \(M_{\mathrm{old}}\) 和 \(M_{\mathrm{new}}\) 针对同一用户属性，产生互斥值。
- 不能有显式否定，不能提及前一个场景。
- 例：“我住在北京” \(\rightarrow\) “我在上海租了房”。

Type II: Propagated Implicit Conflict。

- \(M_{\mathrm{new}}\) 通过改变相关属性，间接否定 \(M_{\mathrm{old}}\)。
- 需要通过依赖链推理检测传递性失效。
- 例：“我每天骑自行车上班” \(\rightarrow\) “我昨天打篮球摔断了腿”。

探测维度：

State Resolution (SR)：

- 显式检测。
- 评估模型是否识别 prior belief 已经不再有效。

Premise Resistance (PR)：

- 前提抵抗。
- 问题包含一个基于 \(M_{\mathrm{old}}\) 的错误前提。
- 评估模型是否识破陷阱，拒绝接受错误前提。

Implicit Policy Adaptation (IPA)：

- 隐式策略适应。
- 需要模型基于新状态做行动计划或推荐。
- 评估最终行为是否遵循 \(M_{\mathrm{new}}\) 而非 \(M_{\mathrm{old}}\)。

整体：

\[
2 \text{ conflict types} \times 3 \text{ probe dimensions} = 6 \text{ evaluation settings}
\]

\[
400 \text{ scenarios} \times 3 \text{ probes} = 1200 \text{ queries}
\]

### 6.3 为什么 STALE 贴合本文主线

STALE 把“前提污染”单独拎出来，这在 memory QA 里常被忽略。

模型能拿到 updated evidence 不等于它会在用户用过期前提提问时拒绝回答。

LightMem 和 mem-0 的 SR / PR 差距说明 retrieval is not enough：你 retrieve 到了对的，但模型还是顺着用户的前提走了。

这说明：

> 新证据被存了、也被取到了，但没有变成回答的支配依据。

这和本文主线一致：

> Retrieval visibility 与 behavioral use 之间有断层。

### 6.4 CUPMem 作为特化 primitive stack

CUPMem 是有效但不够通用的解法。

它的三件事：

- write-side belief updating；
- topology-triggered belief propagation；
- authorized readout。

代码模块对应：

- 写入层：chunker、delta_extractor、invalidation_judge、invalidation_resolver、stale_linker、writer；
- 存储层：typed temporal store、in-memory、transitions、temporal_policy；
- 检索层：embedding；
- 查询层：premise_verifier、basis_recovery、action_grounding、track_hints、support、readout。

这套机制很有针对性，所以分数提升明显，PR 提升尤其大。但它的代价是 schema dependence。CUPMem 是 targeted prototype，依赖 predefined state schema，不是 general-purpose memory architecture；schema-free evolving user state 仍未解决。

这正好说明：

> 一个子任务真正被系统化解决，往往就是先抽出合适的 artifact substrate，再定义一组能操作这个 substrate 的 primitives。

### 6.5 STALE 的形式化解释

STALE 可以自然解释成一个子任务实例：

\[
\text{Subtask} = \text{stale-state validity}
\]

\[
\text{Artifact substrate} = D^{\text{state}}
\]

\[
P_{\text{STALE}} =
\{\rho_{\text{extract}},
\rho_{\text{propagate}},
\rho_{\text{adjudicate}},
\rho_{\text{authorize}},
\rho_{\text{readout}}\}
\]

它做的是：

\[
D_t \rightarrow D_t^{\text{auth}} \rightarrow v_u^{\text{state}} \rightarrow y.
\]

这说明，在 state-validity 子任务上，正确做法可能是先设计 state-temporal artifact，再配套一组能维护和读取 current validity 的 primitives。

### 6.6 STALE 的论文角色

STALE 目前不应过早写成主方法。

它可以作为：

1. Candidate case study：展示 state-validity 子任务如何落入 artifact / primitive / view 框架。
2. Primitive-role alignment 的受控底座：分析哪些原语分别支持 SR、PR、IPA。
3. 解法层的候选：如果能做出下游验证，就回答 “If failure is not merely retrieval, what kind of artifact + primitive stack can fix a concrete subtask?”

需要谨慎：

- 如果只是解释 CUPMem，贡献不够。
- 如果外扩太多，会削弱 ATM-Bench 主线。
- 更稳的路线是让 STALE 支持本文的 state-view / authority-readout 论点，而不是强行作为完整方法贡献。

## 7. Mnemis：理论动机与相关讨论

### 7.1 为什么 Mnemis 重要

Mnemis 的作用不是作为主实验，而是说明 access language / primitive 不是空泛数学。

Mnemis 没有空定义两个算子。它有 base graph 和 hierarchical graph 两个 artifact substrate。

- Base graph 支持 embedding / BM25 / RRF 这类 top-k similarity language。
- Hierarchical graph 支持 top-down browsing language。

它的 System-2 增益来自 hierarchical graph 允许“从类别到实体再到相关 episode / edge”的全局选择，而不是来自一个抽象 System-2 算子。

### 7.2 Mnemis 的结果启发

优势：

- SSP、SSA 接近饱和。
- Single-Hop 很高。
- KU 较高。
- Multi-Hop 有明显提升空间但整体强。

这些任务大多是“找到证据就行”的任务。Single-Hop 和 SSU / SSA 全在高位，说明基础检索精度已经没有问题。

弱势：

- Open-Domain 最低。这不是检索问题，是推理问题：找全证据只是前提，还需要判断或预测。
- Temporal 较弱。System-2 在时间类上弱，因为遍历路径不变，抓不到完整时序链。
- MS / TR 仍有跨 session 或时间推理遗漏。

关键区别：

> Weak = 检索到了但推理不够，还是漏了关键证据？

Open-Domain 更像前者，Temporal 和 MS 更像后者。

### 7.3 System-2 的解释

System-2 带来的增益主要体现在：

1. Multi-Hop 和 Open-Domain。纯相似度容易漏，因为相似度最匹配的不等于最相关。System-2 的层级浏览能找到间接关联的 evidence。
2. Coverage gap 填补。System-2 能为多数查询贡献有效结果，说明纯 System-1 存在覆盖盲区。
3. 具体 case 中，System-1 搜到表面原因，System-2 通过层级浏览找到更根本原因。

System-2 不擅长：

- Temporal 类，因为浏览路径在遍历过程中不变，抓不住完整时序链。
- 需要多次搜索的场景，因为当前 System-2 是一次性自上而下遍历，不能像 multi-step reasoning 那样根据中间结果调整路径。

结论：

> System-2 补的是 coverage 和 recall，不是 precision。Mnemis 赢在“找得更全”，不是“找得更准”。

进一步联系本文：

> 层级浏览能扩大候选证据覆盖，但最终答案仍取决于哪些浏览结果被保留下来并序列化进最终视图。
